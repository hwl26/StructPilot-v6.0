# StructPilot v4 软件体验评估与诊断报告

> 评估对象：`StructPilot_v4`（基于 Streamlit + LangGraph 的 cryo-EM 流程导航 Copilot）
> 评估日期：2026-07-10
> 评估方式：源码级静态走查（main.py / agents/ / graph/ / ui/ / knowledge_base/ / config/）

---

## 一、总体评估结论

| 维度 | 评价 | 说明 |
|---|---|---|
| 功能完整性 | ★★★★☆ | 12 步工作流、双软件、RAG、官方文档集成、报告导出齐全，铁律兼容到位 |
| 架构清晰度 | ★★★★☆ | 配置/资源解析收敛到独立模块，V4 重构质量较好 |
| **LLM 响应速度** | **★★☆☆☆** | 全链路同步阻塞、无流式、单次问答叠 3+ 次网络往返，体感明显偏慢 |
| **回答准确性** | **★★☆☆☆** | 路由为脆弱的关键词顺序匹配，易答非所问；低分 RAG 噪声注入 |
| **交互流畅度/滚动** | **★☆☆☆☆** | 智能自动滚动基本失效（`_chat_auto_scroll` 设而不读），每次 rerun 回顶 |
| 知识库图文能力 | ★★☆☆☆ | 知识库无图（设计如此）；图像问答已支持 OCR+视觉双轨，但可选 vision 钩子默认关 |

**一句话结论**：这是一个功能与架构都相当成熟的领域助手，但**交互层的三处体验短板（慢、答非所问、滚动）会严重拉低“流畅感”**，正好对应你使用中发现的三大痛点。这三者都可修，且改动集中在少数文件，风险可控。

## 二、痛点 1：LLM 回复慢 + 答非所问

### 2.1 「慢」的代码级根因
- **全程同步阻塞，无流式输出**。`main.py:2629` 用 `with st.spinner("🤔 思考中…"): app.handle(...)`，整条链跑完才渲染；LLM/embedding 调用均为同步 `requests.post`（`agents/llm_agent.py:478/692/737`），无 streaming。
- **单次问答叠加 3+ 次网络往返**：
  - `graph/app.py:307` `SmartQAEngine.process()` → `retriever.search` 对 `search_queries[:3]` **循环调用 3 次**；
  - `app.py:352` 再调一次 `retriever.search`（重复检索）；
  - `app.py:392` 再调一次 `llm.rewrite_with_metadata`（1 次 LLM 改写）；
  - 概念路径 `answer_concept`（`smart_qa_engine.py:1662`）额外 2 次检索 + 1 次 LLM。
- **embedding 冷启动极慢**：`retriever._ensure_corpus_ready`（retriever.py:335）首次对整库 embedding（多次同步 HTTP），虽落盘缓存但首跑卡顿；未配 embedding 时退回弱召回的 `_lexical_search`。
- **超时过长**：`llm_agent.py:195` 默认 `timeout=90`，服务商慢时前端最长转 90s；纯同步、无并发。

### 2.2 「答非所问」的代码级根因
- **路由是脆弱的关键词顺序匹配**：`navigator.handle_input`（`navigator_agent.py:240`）靠 `if any(k in text)` 串：plot→sop→concept→expert→fault，**先到先得、并列不加权**。例如“怎么做”易进 sop 而非参数轨道；“怎么看 FSC”进 plot。
- **LLM 意图分类被禁用**：`smart_qa_engine.py:216 understand()` 强制走 `_understand_with_rules`，现成的 `_understand_with_llm`（约 25s）被注释掉；规则计分平局时 `max` 取其一，歧义未处理。
- **LLM 自由发挥无硬约束**：`concept_answer`（`llm_agent.py:601`）在 glossary 未命中、`extra_context=""` 时让模型凭自有知识答，无“不知道就明说”约束；`_system_prompt`（`:420`）只要求“以规则层为准”，但规则层若被错误路由给出无关模板，LLM 只是改写错误内容 → 偏差放大。
- **RAG 低分噪声注入**：`app.py:220 _filter_retrieved`（阈值 `0.35`）仍把低分片段塞进 prompt 当参考，干扰输出。

### 2.3 优化方向（可落地）
- **快**：①改流式（`st.write_stream` + OpenAI streaming），先吐规则层再增量补；②合并重复检索（3 次循环 + polish 检索 → 1 次）；③语料 embedding 启动预热 + 提高 cache 复用；④超时降到 30s，超时即快速降级规则模板。
- **准**：①启用轻量意图分类（或缓存 `_understand_with_llm` 单次分类）替代顺序关键词；②系统提示加“无依据明说不知道”，概念卡强制 glossary 兜底、禁纯自由发挥；③RAG 阈值提到 0.5 并过滤低分噪声；④路由由“首个命中”改为加权打分取最高。

## 三、痛点 2：智能自动滚动实现不佳

### 3.1 当前状态（零散、有死代码、核心标志失效）
- **整页上下跳转按钮**：`main.py:2399 render_scroll_shortcuts()` 仅「↑到顶 / ↓到底」手动双按钮，非智能。
- **聊天面板自动滚到底**：`main.py:3550` 用 `components.html` JS，靠 `window.parent.__sp_user_at_bottom` 判断；仅初始加载/新消息且在底部时滚动。
- **冗余实现 A**：`main.py:3381` 又一份 MutationObserver 版 `scrollToLatest`，与 3550 段功能重复、易冲突。
- **死代码 B**：`utils/image_lazy.py:247 inject_scroll_to_latest_js()` 是复制版，**从未被调用**。
- **死代码 C**：`agent/ui_state_manager.py:239-265` 的 `save/get_scroll_position`（per stage/tab 记忆），**从未被调用**。
- **关键缺陷**：`st.session_state._chat_auto_scroll = True` 在 `main.py:2682/3232/3751` **三处被写入，全代码无任何读取**——该标志纯属摆设。

### 3.2 step 切换 / 预设问题 / 新输入的滚动行为
- **step 切换**：侧边栏 `st.button`（main.py:2765，key=`checkpoint_nav_{cid}`）→ `switch_checkpoint()` 改写 `current_cp_id` → `st.rerun()`（2773）。每次 rerun 整页回顶，只聊天面板 JS 滚到底，外层整页仍停顶部；且 `__sp_scroll_bound` 只绑定一次，rerun 后旧 pane 卸载、监听器指向已脱离节点，底部判定冻结 → 后续滚动时灵时不灵。
- **预设问题**：`_QUICK_QUESTIONS`（main.py ~2960）硬编码字典，`st.button` → `run_command()` → `st.rerun()`（3078），仅向聊天塞文字并 rerun，**无针对该问题的定位滚动**。
- **新文字/图像**：`main.py:3739 st.chat_input` → `run_command()` → 置 `_chat_auto_scroll=True`（死标志）→ rerun；因消息数变化聊天底部 JS 触发滚底，但同样受监听器失效影响，且整页回顶。

### 3.3 智能自动滚动应如何实现
- **统一为单一** `components.v1.html` 注入：用 `st.query_params` 或隐藏 `st.markdown('id="cp_xx"')` 锚点标记当前 step；rerun 后 JS 按触发源选目标——切换 step/提交提问 → `scrollIntoView` 到该 step 工作区或聊天底；点击预设问题 → 滚到对应聊天气泡。
- **每次 rerun 重新绑定** `MutationObserver`（不要 `__sp_scroll_bound` 一次性开关），避免旧节点失效。
- **保留“用户上滑读历史不抢滚”**：用局部变量记录 at-bottom，而非挂在 `window.parent` 的永久状态。
- **整页定位**：step 切换后用 `location.hash` / `<a href="#anchor">` 让整页定位到工作区而非顶部。

## 四、痛点 3：知识库里有没有图？为什么没有？

**结论：知识库（knowledge_base/）里没有任何图片文件。** 经 `Glob knowledge_base/**/*` 确认，全部为 32 个 `.json`、21 个 `.md`、9 个 `.py`、4 个 `.jsonl` 等结构化文本，**零 `.png/.jpg/.webp`**。

**为什么没有（设计如此，非 bug）：**
- 知识库定位是“规则 / SOP / 故障库”，刻意只存结构化文本（pipeline_checkpoints.json、fault_trouble.json、tier2_rules.json、sop_template.json 等）与 Markdown。
- 截图走**另一条外置通道**：`config/screenshot_map.py` 把 `cp_NN` 映射到**上级目录 `StructPilot_Visual_UI/images`**（V4 刻意不把截图复制进工程），缺失时回退 `assets/guides`。**当前沙箱里该上级目录也不存在**，故 UI 图库为空。
- 知识库唯一与图相关的入口是管理页“图文操作文档导入”（`main.py:4059` 的 `op_images` 上传器），但它**生成文本草稿写入 KB**，图片本身不落进 knowledge_base 目录。

> 也就是说：不是“图丢了”，而是“知识库本来就不含图；截图依赖外部目录且当前未就位”。要让 KB 有图，需明确产品决策：是把截图纳入 KB 管理，还是维持外置映射并补上图片源。

## 五、痛点 4：问答时对图的处理流程，是 OCR 吗？

**结论：不是单纯的 OCR——是「本地 OCR 审计 + 原图直传视觉模型」双轨。**

1. **上传入口**：主聊天 `st.chat_input(..., accept_file="multiple")`（main.py:3739）→ `save_uploaded_images`（720）→ `image_refs` → `state.pending_images`（2587）。
2. **本地 OCR / 规则提取**：`infer_image_observations`（1024）→ `run_local_ocr`（938，用 `pytesseract`）提取参数，再做可审计的仲裁 `arbitrate_image_params`。这是**无需外部 API、可溯源**的默认通道（回答里会显示 OCR 文本摘录）。
3. **原图直传视觉模型**：`graph/app.py:379` 取 `image_paths`，`:392` 调 `rewrite_with_metadata(image_paths=...)`。`agents/llm_agent.py` 实际把图当多模态输入：
   - OpenAI-compatible：`_openai_compatible_rewrite`（459-467）拼 `image_url` data URL；
   - Anthropic（538-570）与 Gemini（572-589）均为 base64 内联图像——**README 说“Anthropic/Gemini 多模态为后续工作”已过时**，代码其实已实现。
4. **可选 vision hook**：`run_optional_vision_model`（955）受 `STRUCTPILOT_ENABLE_VISION` 控制，**默认关闭**，仅作 advisory，不参与关键参数判定。

> 一句话：当前主聊天问答中，你上传的图**既被 pytesseract 做 OCR 提取参数（可审计），也被原样以多模态发给视觉 LLM 直接“看图”**（默认主走 OpenAI-compatible 视觉模型，Anthropic/Gemini 也已落地）。独立 `STRUCTPILOT_ENABLE_VISION` 钩子才是未启用的预留项。
